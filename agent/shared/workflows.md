# EO Studio - Common Workflows

이 문서는 모든 에이전트가 참조하는 공통 워크플로우를 정의합니다.

## 1. 알림 워크플로우 (Notification Workflow)

### 알림 우선순위 및 채널

#### 🔴 긴급 (Critical)
**채널**: Slack DM (즉시)
**대상**: Seohyun Ahn
**상황**:
- 캐시 부족 ($10,000 이하)
- 90+ days overdue AR 항목
- 법률 긴급 이슈 (소송, 계약 위반 등)
- 시스템 다운 (주요 API 3회 연속 실패)
- 데이터 무결성 이슈

**알림 포맷**:
```
🚨 CRITICAL ALERT

Issue: [문제 요약]
Impact: [영향 범위]
Status: [현재 상태]
Action Required: [필요한 조치]
Deadline: [기한 있으면]

Details: [상세 내용]
```

#### 🟡 중요 (Important)
**채널**: Slack 채널 알림
**대상**: 관련 팀 채널
**상황**:
- AR overdue 증가 (전주 대비 20% 이상)
- 계약 만료 임박 (D-30)
- 주간 리포트
- FX 급변동 (threshold 초과)
- 시스템 성능 저하

**알림 포맷**:
```
⚠️ IMPORTANT UPDATE

Topic: [주제]
Summary: [요약]
Impact: [영향]
Next Steps: [다음 단계]
```

#### 🟢 정보 (Info)
**채널**: Slack 채널 (비 멘션) 또는 Notion
**대상**: 해당 프로젝트 팀
**상황**:
- 일일 리포트
- 일반 업데이트
- 시스템 로그
- 완료된 작업 요약

**알림 포맷**:
```
ℹ️ Daily Update

Date: [날짜]
Summary: [요약]

Key Metrics:
- [지표 1]: [값]
- [지표 2]: [값]

Details: [Link to Notion]
```

### Slack 채널 매핑

```
#finance           → 재무 관련 알림
#legal             → 법무 관련 알림
#operations        → 운영 관련 알림
#general           → 전체 공지
#dev-alerts        → 시스템/기술 알림
```

## 2. 리포팅 워크플로우 (Reporting Workflow)

### 리포트 유형 및 주기

#### 일일 리포트 (Daily Report)
**생성 시간**: 매일 21:00 UTC (KST 06:00 다음날)
**담당 에이전트**: 각 프로젝트 에이전트
**저장 위치**: `agent/projects/{project}/reports/daily/`
**발송 채널**: Slack (해당 채널)

**포함 내용**:
- 오늘 완료된 작업
- 주요 이벤트/이슈
- 다음 날 계획
- Key metrics (프로젝트별)

#### 주간 리포트 (Weekly Report)
**생성 시간**: 매주 월요일 09:00 UTC (KST 18:00)
**담당 에이전트**: 각 프로젝트 에이전트
**저장 위치**: `agent/projects/{project}/reports/weekly/`
**발송 채널**: Slack + Notion

**포함 내용**:
- 지난 주 성과 요약
- 주요 metrics 및 트렌드
- Blockers 및 리스크
- 다음 주 우선순위

#### 월간 리포트 (Monthly Report)
**생성 시간**: 매월 1일 09:00 UTC (KST 18:00)
**담당 에이전트**: 각 프로젝트 에이전트 + 통합
**저장 위치**: `agent/projects/{project}/reports/monthly/`
**발송 채널**: Notion (Executive Dashboard)

**포함 내용**:
- 월간 성과 요약
- 예산 대비 실적
- 주요 이벤트 및 결정 사항
- 다음 달 계획 및 목표

### 리포트 포맷 표준

모든 리포트는 다음 구조를 따름:

```markdown
# [Report Type] - [Date]

## Executive Summary
[2-3 문장으로 핵심 요약]

## Key Metrics
- Metric 1: [값] ([변화])
- Metric 2: [값] ([변화])
- Metric 3: [값] ([변화])

## Highlights
- ✅ [성과 1]
- ✅ [성과 2]
- ⚠️ [주의 사항 1]

## Details
[상세 내용]

## Action Items
- [ ] [액션 1] (@담당자, 기한)
- [ ] [액션 2] (@담당자, 기한)

## Next Steps
[다음 단계]

---
Generated by: [Agent Name]
Date: [Timestamp]
```

## 3. 데이터 동기화 워크플로우 (Data Sync Workflow)

### 동기화 대상 시스템

```
Bill.com ←→ Notion (AR 데이터)
ClickUp ←→ Notion (프로젝트 데이터)
Gmail ←→ Notion (중요 이메일 로그)
Plaid ←→ Google Sheets (은행 거래)
Google Drive ←→ Notion (문서 링크)
```

### 동기화 주기

- **실시간**: Bill.com → Notion (AR 매칭 시)
- **매시간**: Plaid → Notion (은행 잔액)
- **일일**: ClickUp → Notion (프로젝트 상태)
- **주간**: 전체 데이터 무결성 체크

### 동기화 실패 시 처리

1. 3회 자동 재시도 (1분 간격)
2. 재시도 실패 시 Slack 알림 (#dev-alerts)
3. 에러 로그 저장 (`logs/sync_errors.log`)
4. 수동 개입 필요 시 Seohyun에게 DM

## 4. 승인 워크플로우 (Approval Workflow)

### 자동 승인 (Auto-Approval)

다음 작업은 자동으로 실행 (사전 승인):
- ✅ 일일/주간/월간 리포트 생성
- ✅ Confidence 80% 이상 AR 매칭
- ✅ 알림 발송
- ✅ 데이터 조회 및 읽기
- ✅ 리포트 아카이브

### 수동 승인 필요 (Manual Approval Required)

다음 작업은 반드시 확인 필요:
- 🔒 Bill.com 인보이스 상태 업데이트
- 🔒 재무 데이터 수정
- 🔒 법률 문서 변경
- 🔒 대외 이메일 발송
- 🔒 계약서 서명
- 🔒 $1,000 이상 결제
- 🔒 주주명부 수정

### 승인 요청 프로세스

1. 에이전트가 Slack DM으로 승인 요청
2. 요청 내용:
   - 작업 설명
   - 이유 (Why)
   - 영향 (Impact)
   - 롤백 가능 여부
3. 승인 버튼 또는 텍스트 응답
4. 승인 후 실행 및 결과 알림

## 5. 에러 처리 워크플로우 (Error Handling Workflow)

### 에러 분류

#### Level 1: Info
- 로그만 기록
- 자동 복구 가능
- 예: Rate limit (자동 대기 후 재시도)

#### Level 2: Warning
- 로그 기록 + Slack 알림 (#dev-alerts)
- 자동 복구 시도
- 예: API timeout (3회 재시도)

#### Level 3: Error
- 로그 기록 + Slack 알림 (담당자 멘션)
- 수동 개입 필요
- 예: 데이터 무결성 오류

#### Level 4: Critical
- 로그 기록 + Slack DM (즉시)
- 즉시 수동 개입 필요
- 시스템 일부 정지 가능
- 예: 데이터 손실 위험, 보안 이슈

### 에러 복구 프로세스

```
1. 에러 감지
2. 에러 레벨 판단
3. 자동 복구 시도 (Level 1, 2)
   ↓ 실패
4. 알림 발송 (Level에 따라)
5. 수동 개입 대기
6. 복구 후 재시작
7. Post-mortem (Critical 에러만)
```

## 6. 백업 및 아카이브 (Backup & Archive)

### 백업 대상

```
agent/projects/*/memory/*.md       → 매일 백업
agent/projects/*/reports/*         → 영구 보관
logs/*.log                         → 30일 보관
.env                              → 암호화 백업 (수동)
```

### 백업 위치

- **로컬**: `backups/` 디렉토리
- **클라우드**: Google Drive (자동 동기화)
- **Git**: agent/ 디렉토리는 git commit

### 보관 정책

- **일일 리포트**: 90일 보관
- **주간 리포트**: 1년 보관
- **월간 리포트**: 영구 보관
- **로그 파일**: 30일 보관
- **메모리 파일**: 영구 보관 (git)

## 8. 에이전트 간 협업 (Agent Collaboration)

### 협업이 필요한 시나리오

#### 시나리오 1: AR 매칭 완료 후
```
AR Agent → Finance Agent
- AR 매칭 완료 알림
- 업데이트된 캐시플로우 데이터 공유
```

#### 시나리오 2: 계약 만료 임박
```
Legal Agent → Finance Agent → Operations Agent
- Legal: 계약 만료 30일 전 알림
- Finance: 예산 영향 분석
- Operations: 갱신 태스크 생성
```

#### 시나리오 3: 주간 리포트 통합
```
All Agents → Operations Agent
- 각 에이전트가 주간 리포트 생성
- Operations가 통합 리포트 작성
- Notion에 발행
```

### 협업 프로토콜

1. **명확한 데이터 교환**: JSON 또는 Markdown 포맷
2. **알림 표준 준수**: 위의 알림 포맷 사용
3. **Notion을 중앙 저장소로**: 모든 데이터는 Notion 동기화
4. **중복 작업 방지**: 각 에이전트의 역할 명확히

## 9. 모니터링 및 헬스 체크 (Monitoring & Health Check)

### 시스템 헬스 지표

```
✅ Healthy:
- API response time < 2초
- Error rate < 1%
- 모든 일일 리포트 정상 발송
- 데이터 동기화 지연 < 5분

⚠️ Warning:
- API response time 2-5초
- Error rate 1-5%
- 리포트 지연 < 1시간
- 데이터 동기화 지연 5-30분

🔴 Critical:
- API response time > 5초
- Error rate > 5%
- 리포트 미발송
- 데이터 동기화 실패 > 30분
```

### 모니터링 체크리스트

#### 일일 체크
- [ ] 모든 API 연결 정상
- [ ] 일일 리포트 발송 완료
- [ ] 에러 로그 검토 (Critical/Error만)
- [ ] 데이터 동기화 확인

#### 주간 체크

- [ ] 성능 지표 검토
- [ ] 백업 상태 확인
